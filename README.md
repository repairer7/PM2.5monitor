# PM2.5 高值站点监控

一个面向生态环境系统大气污染防治工作人员的 PM2.5 高值站点即时监控工具。

项目通过中国环境监测总站空气质量发布接口获取最新国控站点空气质量数据，自动归档完整监测数据，并针对苏州市计算各站点 PM2.5 浓度与全市站点平均值的差异。当某个站点明显高于全市平均水平时，系统通过 Bark 将高值站点信息推送到手机，帮助工作人员及时掌握污染异常点位，为污染过程研判、现场巡查、执法检查和应急响应提供快速线索。

## 适用场景

本项目适用于生态环境系统从事大气污染防治、环境监测、污染防控和应急值守的工作人员，尤其适合以下工作场景：

- 值班期间快速掌握区域内 PM2.5 突出站点
- 污染天气过程中及时发现局地高值区域
- 开展重点区域、重点时段的大气污染巡查
- 为污染成因分析和传输影响研判提供站点线索
- 辅助安排走航监测、现场核查和执法检查
- 通过手机推送减少反复登录平台查看数据的操作成本

需要注意，本工具用于快速发现异常和高值线索，不替代环境监测平台的正式数据审核、污染成因分析和行政决策流程。

## 核心功能

- 获取中国环境监测总站发布的最新空气质量站点数据
- 提取时间点、站点编码、站点名称、区域、经纬度和主要污染物等字段
- 保存 PM2.5、PM10、臭氧、二氧化氮、二氧化硫、一氧化碳等监测数据
- 按日期和监测时间自动归档 CSV 文件
- 自动筛选苏州市站点
- 计算苏州市站点 PM2.5 平均值
- 筛选高于全市平均值 40 的站点
- 通过 Bark 推送高值站点名称、PM2.5 浓度和偏离平均值的差值
- 当没有高值站点时不发送通知
- 当同一批数据出现多个时间点时，额外保存异常数据，便于排查接口或数据质量问题

## 高值站点判定规则

当前代码采用相对高值判定方式，而不是直接按照国家标准限值判断：

```text
苏州市 PM2.5 平均值 = 所有苏州市站点 PM2.5 的平均值
站点差值 Δ = 站点 PM2.5 - 苏州市 PM2.5 平均值
高值站点 = Δ > 40
```

推送内容示例：

```text
某某站 PM2.5=86.0 Δ=43.2
```

其中：

- `PM2.5`：该监测站点的 PM2.5 数据
- `Δ`：该站点相对于苏州市站点平均值的偏高幅度
- `Δ > 40`：当前项目的高值站点触发阈值

该规则适合快速识别区域内部的突出站点。实际开展污染成因分析时，还应结合风向风速、边界层、湿度、其他污染物、周边排放源和站点数据质量情况综合判断。

## 数据来源

项目当前访问中国环境监测总站空气质量发布接口：

```text
https://air.cnemc.cn:18007/HourChangesPublish/GetAllAQIPublishLive
```

程序获取接口返回数据后，会统一处理字段名称和时间格式，并将缺失标记 `—` 转换为空值。

## 数据归档

监测数据按照以下结构保存：

```text
data/
└── YYYY-MM-DD/
    └── YYYY-MM-DDTHH.csv
```

每次运行会将当前时间点的数据追加到对应 CSV 文件中。归档字段包括：

- 监测时间
- 站点编码
- 经度和纬度
- 区域和站点名称
- 首要污染物
- AQI
- PM2.5、PM10、臭氧、二氧化氮、二氧化硫和一氧化碳
- 对应的 24 小时或 8 小时统计值

这些归档数据可用于后续开展：

- 污染过程回溯
- 站点之间对比
- 高值站点变化趋势分析
- 污染时段和区域分析
- 现场核查后的数据复盘

## 使用 GitHub Actions

### 1. Fork 仓库

点击 GitHub 仓库右上角的 **Fork**，将项目复制到自己的 GitHub 账号下。

### 2. 配置 Bark 推送

进入仓库：

```text
Settings → Secrets and variables → Actions → New repository secret
```

添加以下 Secrets：

| 名称 | 说明 | 示例 |
| --- | --- | --- |
| `BARK_SERVER` | Bark 服务地址或域名 | `api.day.app` |
| `BARK_KEY` | Bark 设备 Key | `your-bark-key` |

代码会自动拼接以下格式的推送地址：

```text
https://BARK_SERVER/BARK_KEY/标题/内容
```

请不要将 Bark Key 直接写入代码或提交到公开仓库。

### 3. 手动运行监控

进入仓库的 **Actions** 页面，选择：

```text
Air Quality Monitor
```

点击：

```text
Run workflow
```

工作流会依次完成：

1. 检出代码
2. 安装 Python 3.11
3. 安装项目依赖
4. 执行 `PM2.5monitor.py`
5. 将新增的 `data/` 数据提交并推送回仓库

当前工作流使用 `workflow_dispatch`，默认只能手动触发，并没有配置自动定时任务。

## 本地运行

### 环境要求

- Python 3.9 或更高版本
- 能够访问中国环境监测总站接口的网络环境
- Bark 设备和有效的 Bark Key

### 安装依赖

```bash
python -m pip install --upgrade pip
pip install -r requirements.txt
```

### 配置环境变量

Linux/macOS：

```bash
export BARK_SERVER="api.day.app"
export BARK_KEY="your-bark-key"
```

Windows PowerShell：

```powershell
$env:BARK_SERVER="api.day.app"
$env:BARK_KEY="your-bark-key"
```

运行程序：

```bash
python PM2.5monitor.py
```

运行结果包括：

- 控制台输出数据归档路径
- 是否发现苏州市 PM2.5 高值站点
- Bark 推送结果
- 新增的 CSV 监测数据

## 修改监控区域

当前脚本固定筛选：

```python
df.area == '苏州市'
```

如果需要监控其他地市，可修改为目标区域名称：

```python
df.area == '目标城市'
```

如果需要同时监控多个区域，可以将筛选逻辑扩展为多个区域分别计算平均值和高值站点，避免不同城市之间的浓度水平相互影响。

## 修改高值阈值

当前高值站点阈值为高于区域平均值 40：

```python
high_df = df2[df2['diff'] > 40]
```

可以根据实际业务需求调整阈值。例如：

```python
high_df = df2[df2['diff'] > 30]
```

调整阈值时，建议结合站点数量、区域污染水平、预警响应要求和历史数据进行评估，避免阈值过低造成频繁提醒，或阈值过高导致遗漏需要关注的站点。

## Bark 通知

通知标题为：

```text
出现高值站点
```

通知分组为：

```text
大气站点监控
```

当存在高值站点时，通知正文按行列出站点名称、PM2.5 数值和偏离区域平均值的差值。当没有高值站点时，程序只在日志中输出“无高值站点”，不会发送空通知。

推送失败时，程序最多重试 3 次，等待时间依次为 5 秒和 10 秒。多次失败后会放弃本次推送，但已经获取和归档的数据仍会保留。

## 面向大气污染防治工作的使用建议

建议将该项目部署在承担大气污染防治值守或研判任务的 GitHub 账号中，并根据工作时段配置自动执行。每次运行后，工作人员可以优先关注推送中的高值站点，再结合以下信息开展研判：

- 站点 PM2.5 与区域平均值的差值
- 高值站点的经纬度和空间分布
- PM10、臭氧、二氧化氮等协同污染物
- 风向、风速和污染物传输方向
- 高值站点周边工业企业、施工工地、道路和扬尘源
- 站点历史趋势及相邻站点变化
- 现场巡查、走航监测和企业排放情况

该工具更适合作为“即时发现高值线索”的辅助工具：先通过手机通知缩短发现时间，再由业务人员结合正式监测数据和现场信息进行核实和处置。

## 注意事项

- 当前只筛选苏州市，其他区域需要修改代码
- 当前高值判定是相对区域平均值的差值判断，不等同于国家空气质量标准超标判断
- 监测接口可能发生调整、限流或暂时不可用
- 程序使用 `verify=False` 访问接口，生产环境中应根据网络和证书条件评估安全风险
- 接口返回缺失值 `—` 时会被转换为空值，平均值计算结果应结合有效站点数量判断
- 同一批数据包含多个时间点时，程序会写入 `data/Error/` 目录供排查
- GitHub Actions 需要 `contents: write` 权限才能自动提交归档数据
- 当前工作流没有自动定时配置，需要手动运行或自行增加 `schedule`
- 请遵守中国环境监测总站网站的访问规范和相关服务要求
- 归档数据和推送信息应按照本单位数据管理、信息安全和业务使用要求进行管理
